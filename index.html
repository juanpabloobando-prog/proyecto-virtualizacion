Proyecto de Virtualización

VirtualBox · Docker · Kubernetes · GitHub Pages

El proyecto tiene como objetivo explorar y aplicar conceptos fundamentales de virtualización e infraestructura moderna de software, mediante la implementación práctica de diferentes tecnologías en entornos controlados. A lo largo de los cuatro componentes se abordaron temáticas como la configuración de máquinas virtuales con VirtualBox sobre Fedora 44, administración mediante SSH, orquestación de contenedores con Kubernetes usando Minikube, y la documentación del proceso mediante un sitio web publicado.

Desarrollo del Proyecto

COMP 01

Virtualización con Linux

Esta componente consiste en la instalación de dos VMs en VirtualBox: la primera con Fedora 44 Workstation (entorno gráfico) y la segunda con Fedora 44 Server (solo consola). Se configuraron particiones manuales, se estableció la red entre ambas máquinas y se activó el acceso SSH desde la VM gráfica hacia la VM consola.

Instalación — Fedora 44 Workstation

En esta captura se evidencia el proceso de instalación de Fedora 44 en su edición Workstation. Se puede ver el progreso con la etapa de instalación de software en curso, confirmando que el sistema fue instalado correctamente en la VM de entorno gráfico.

Particionamiento — Fedora 44 Workstation

En esta captura se evidencian las particiones de la VM Workstation mediante el comando lsblk. Aunque el instalador gráfico no permitió el particionamiento manual durante la instalación, se puede observar la estructura resultante con las particiones /boot/efi, /boot, y los volúmenes lógicos para /, swap y /home.

Particionamiento Manual — Fedora 44 Server

En esta captura se evidencia que, a diferencia del entorno gráfico, la instalación de Fedora Server sí permitió realizar el particionamiento manual. Se configuraron 4 particiones: /boot/efi, /boot, / y /home, cumpliendo con el requisito mínimo del proyecto.

Configuración de Red — Fedora 44 Workstation

En la siguiente captura se muestra la configuración de red de la VM Workstation a través de la terminal del entorno gráfico. Se evidencian las direcciones IP de cada interfaz: la red NAT (enp0s3) con IP dinámica y la red solo-anfitrión (enp0s8) con IP estática en 192.168.100.10, ambas en estado activo (UP).

Configuración de Red — Fedora 44 Server

En esta captura se muestra la misma configuración de red pero vista desde la consola de la VM Server. Los datos esenciales como las IPs de las interfaces y sus estados activos se mantienen, confirmando que ambas máquinas están correctamente configuradas en la misma red solo-anfitrión.

SSH Activo en VM Server

En la siguiente captura se muestra el estado del servicio SSH en la VM Server a través de la terminal, donde podemos verificar que el servicio sshd se encuentra activo y en ejecución (active running). Además, en la parte inferior se pueden observar los registros de conexiones SSH exitosas realizadas desde la VM Workstation hacia el Server, confirmando que la comunicación entre ambas máquinas virtuales funciona correctamente.

Conexión SSH entre VMs

En la siguiente captura se muestra la conexión SSH realizada desde la VM Workstation hacia la VM Server, donde se ejecuta el comando ssh juan@192.168.100.20 y se ingresa la contraseña correspondiente. Una vez autenticado, el sistema confirma el acceso exitoso mostrando el último inicio de sesión registrado, lo que evidencia que la comunicación entre ambas máquinas virtuales a través de la red solo-anfitrión está funcionando correctamente.

Diagrama de Arquitectura

Representación visual de la arquitectura implementada, mostrando la relación entre las dos VMs y la comunicación SSH a través de la red Host-Only.

COMP 02

Contenedores Docker

En esta componente se instaló Docker Engine en la VM Workstation y se crearon Dockerfiles propios para un servidor frontend con Nginx y un servidor backend con Python. Los servicios se orquestaron mediante docker-compose con volúmenes persistentes para el frontend.

Dockerfile — Frontend (Nginx)

En esta captura se muestra el contenido del Dockerfile creado para el servicio frontend. Está basado en la imagen de Fedora 44, instala Nginx, copia el archivo index.html al directorio de publicación del servidor y lo expone en el puerto 80, levantando Nginx en modo primer plano.

Dockerfile — Backend (Python)

En esta captura se muestra el Dockerfile del servicio backend. Basado igualmente en Fedora 44, instala Python 3, define el directorio de trabajo en /app, copia el script server.py y expone el puerto 5000, ejecutando el servidor Python directamente al iniciar el contenedor.

Archivo docker-compose.yml

En esta captura se muestra el archivo docker-compose.yml que orquesta ambos servicios. El frontend se construye desde su directorio, se mapea al puerto 80 del host y cuenta con un volumen persistente para los archivos HTML. El backend se construye desde su propio directorio y se expone en el puerto 5000.

Servicios funcionando en el navegador

En las siguientes capturas se evidencia el acceso a ambos servicios desde el navegador dentro de la VM, confirmando que tanto el frontend como el backend están activos y respondiendo correctamente en sus respectivos puertos.

Frontend — localhost:80
Backend — localhost:5000

COMP 03

Orquestación con Kubernetes

En esta componente se instaló Minikube en la VM Workstation y se desplegó un servidor Nginx con múltiples réplicas usando manifiestos YAML. El servicio se expuso mediante NodePort y se demostró el escalado horizontal del despliegue.

Deployment con 2 réplicas y servicio NodePort

En esta captura se muestra el resultado de los comandos kubectl get pods y kubectl get svc, donde se evidencia que el deployment de Nginx tiene 2 pods corriendo correctamente en estado Running, y que el servicio está expuesto como NodePort en el puerto 30080.

Escalado horizontal a 3 réplicas

En esta captura se muestra el escalado manual del deployment a 3 réplicas mediante el comando kubectl scale deployment nginx --replicas=3. Inmediatamente después, se verifica con kubectl get pods que los 3 pods están activos y en estado Running, confirmando el escalado exitoso.

Acceso a Nginx desde el navegador vía NodePort

En esta captura se evidencia el acceso al servidor Nginx desplegado en Kubernetes desde el navegador. Utilizando la IP del clúster obtenida con minikube ip (192.168.49.2) y el puerto NodePort 30080, se confirma que el servicio es accesible externamente y que Nginx responde correctamente.

Diagrama de Arquitectura

Representación visual del clúster de Minikube con el Deployment de Nginx, las 3 réplicas en ejecución y el Service NodePort para acceso externo.

Conclusiones

A lo largo del proyecto se integraron habilidades de administración de sistemas, redes, infraestructura como código y documentación técnica, reflejando un flujo de trabajo cercano al que se aplica en entornos reales de DevOps e infraestructura en la nube. Cada componente aportó una perspectiva distinta sobre cómo las tecnologías de virtualización y contenedores resuelven problemas reales de escalabilidad, portabilidad y gestión de servicios.El proyecto permitió integrar conocimientos de virtualización, contenedores y orquestación de servicios, fortaleciendo habilidades prácticas en administración de sistemas Linux y despliegue de aplicaciones utilizando herramientas ampliamente empleadas en entornos profesionales.
